# Project Layout

This document explains the directory structure of the Pine Tar Sports Fund Deck Builder, where everything lives, and how the pieces fit together.

## Top-Level Files

| File | Purpose |
|------|---------|
| `index.html` | Vite entry HTML |
| `vite.config.ts` | Vite configuration (sets `base: "/pinetarsportsfund/"` for GitHub Pages) |
| `tsconfig.json` | Root TypeScript config (references app + node configs) |
| `tsconfig.app.json` | TypeScript config for the app source (`src/`) |
| `tsconfig.node.json` | TypeScript config for Node-side files (Vite config, server, etc.) |
| `eslint.config.js` | ESLint flat config with React hooks and refresh plugins |
| `postcss.config.js` | PostCSS config (Tailwind CSS v4 + Autoprefixer) |
| `package.json` | Dependencies and scripts |
| `pnpm-lock.yaml` | Primary lockfile generated by pnpm (authoritative dependency graph) |
| `pnpm-workspace.yaml` | pnpm workspace settings |
| `ProjectLayout.md` | This project structure reference document |

## Feature Module Convention

Each feature under `src/features/` follows this structure where applicable:

| Folder | Contents |
|--------|----------|
| `model/` | TypeScript types and Zod schemas |
| `routes/` | Route-level page components (default exports) |
| `context/` | React context providers and hooks |
| `utils/` | Pure utility functions |
| `components/` | Feature-specific UI components |
| `api/` | API call functions (when backend exists) |
| `index.ts` | Barrel file re-exporting the public API |

Not every feature needs all folders вЂ” only add what is actually used.

## Routing

Routes are defined in `src/App.tsx`. The app uses tenant-aware routing: every authenticated route lives under a resolved `TenantProvider` context.

### Tenant resolution

| Method | Pattern | Example |
|--------|---------|---------|
| Path prefix | `/t/:tenantSlug/*` | `/t/pinetarsportsfund/decks` |
| Hostname subdomain | `{slug}.example.com` | `pinetarsportsfund.example.com` |
| Fallback | hardcoded `pinetarsportsfund` | `localhost`, `*.github.io` |

### Route table

| Path | Component | Auth | Description |
|------|-----------|------|-------------|
| `/login` | `LoginPage` | No | Login form |
| `/` | `DashboardPage` | Yes | Main dashboard |
| `/decks` | `DeckListPage` | Yes | List all decks |
| `/decks/new` | `DeckFormPage` | Yes | Create a new deck |
| `/decks/:deckId/edit` | `DeckFormPage` | Yes | Edit an existing deck |
| `/decks/:deckId/preview` | `DeckPreviewPage` | Yes | Preview deck in browser |
| `/exports/:deckId` | `ExportPage` | Yes | Generate export outputs |
| `/admin` | `AdminDashboardPage` | Yes | Admin panel |
| `/settings` | `SettingsPage` | Yes | Theme and preferences |
| `/view/:slug` | `DeckPreviewPage` (isPublic) | No | Public shareable deck view |

All authenticated routes also work under `/t/:tenantSlug/*` for explicit tenant scoping.

## Data Layer

The app supports two data modes:

### In-browser SQLite (default вЂ” no server needed)

`src/lib/api/mock/sqlite.ts` powers an in-memory SQLite database initialized with `sql.js` and persisted to IndexedDB via `localforage`. Tables: `decks`, `assets`, `financial_models`.

The mock API modules (`decks.ts`, `assets.ts`, `financials.ts`, `tenants.ts`) wrap this SQLite layer behind stable CRUD interfaces.

- `data.ts` вЂ” seed data for decks, templates, financial defaults
- `db.ts` вЂ” re-exports low-level SQLite table helpers
- `tenants.ts` вЂ” tenant lookup (returns the `pinetarsportsfund` default tenant)

### Express API server (optional)

`server/index.mjs` is a minimal Express app that exposes REST endpoints for decks, assets, and financial models backed by a SQLite file at `data/app.sqlite.bin`.

```
GET/POST        /api/decks
GET/PUT/DELETE  /api/decks/:id
GET/POST        /api/assets
DELETE          /api/assets/:id
GET/POST/PUT/DELETE  /api/financial-models/:id
```

Start it with `pnpm api` or run alongside Vite with `pnpm dev:full`. Set `VITE_API_BASE_URL=http://localhost:8787/api` to point the frontend at it.

`src/lib/api/http.ts` is the shared HTTP client (`apiRequest<T>`) used when talking to the real server.

## Styling

- **Tailwind CSS v4** is used for all styling
- Dark mode is implemented via a `theme` state in `AppShell` and passed through `useOutletContext`
- Brand styling (primary color, accent color, typography rules) can be applied via context and inline styles after WCAG contrast checking

## Key Patterns

- **Zod validation**: Form data is validated with `safeParse()` and surfaced to the UI in a controlled way
- **Schema-first content modeling**: Decks, slide sections, templates, and financial inputs are strongly typed before rendering or export
- **Collection IDs**: Deck sections, assets, and template blocks can use `crypto.randomUUID()` for client-side IDs
- **Async cleanup**: `useEffect` hooks use cancellation flags (`let cancelled = false`) to prevent state updates after unmount
- **Export separation**: UI editing and output generation stay separated so PPTX/PDF builders can evolve without destabilizing the editor
